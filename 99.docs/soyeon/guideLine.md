# 시스템 구축 보안 가이드

---

## 1. Web & API 보안 (최우선 순위)

### 🚨 API Server 및 관리 콘솔 TLS/SSL 암호화 적용 (HTTPS)
* **설명**: Gitlab API, ArgoCD, K8s API Server 등 외부 및 내부 통신 시 평문 통신을 차단하고 데이터를 암호화합니다.
* **설정 방법**: Nginx Ingress Controller 또는 AWS ALB에 SSL/TLS 인증서(ACM, Let's Encrypt)를 적용하고, 80 포트 요청을 443으로 Redirect 설정합니다.
* **공격 가능성**: 평문 통신 시 중간자 공격(MITM)을 통해 관리자 세션 쿠키, API 토큰, 소스코드 등이 스니핑되어 인프라 전체 권한이 탈취될 수 있습니다.
* **진단 기준**: 모든 관리 API 엔드포인트 호출 시 HTTPS 프로토콜만 허용하고 HTTP 요청은 거부하거나 리다이렉트해야 합니다.
* **진단 방법(예시)**: `curl -Iv example.com` 명령을 실행하여 `301 Moved Permanently` 또는 거부 응답이 떨어지는지 확인합니다.
* **기반시설기준항목**: 서비스 관리 - 웹 서비스 정보 보호

### 🔑 API 인증 토큰 및 Secret 관리
* **설명**: API 접근을 위한 토큰, AccessKey, DB 패스워드를 코드에 하드코딩하지 않고 안전하게 격리합니다.
* **설정 방법**: Kubernetes Secret을 사용하거나 AWS Secrets Manager, HashiCorp Vault와 연동합니다. Git에 푸시하기 전 `.gitignore`를 철저히 관리하고 AWS 내부에서는 IAM Role(IRSA)을 사용합니다.
* **공격 가능성**: GitHub 등 외부 저장소에 토큰이 유출될 경우, 공격자가 자원 채굴(클라우드 마이닝)에 악용하거나 내부 인프라에 침투할 수 있습니다.
* **진단 기준**: 소스코드 및 형상관리 이력 내에 평문으로 된 크리덴셜(비밀번호, 토큰)이 존재하지 않아야 합니다.
* **진단 방법(예시)**: 깃랩 리포지토리에서 `trufflehog` 또는 `gitleaks` 툴을 구동하여 하드코딩된 비밀키가 탐지되는지 스캔합니다.
* **기반시설기준항목**: 시스템 보안 - 사용자 비밀번호 관리 및 접근 통제

### 🛡️ Rate Limiting (과도한 API 요청 차단)
* **설명**: 특정 IP나 사용자가 짧은 시간 동안 과도한 API 요청을 보내는 것을 제한하여 시스템을 보호합니다.
* **설정 방법**: Nginx Ingress 설정에 `nginx.ingress.kubernetes.io/limit-rps` 어노테이션을 추가하거나 AWS WAF를 연동하여 임계치를 설정합니다.
* **공격 가능성**: API 무차별 대입 공격(Brute Force)으로 계정이 탈취되거나, DDoS 공격으로 인해 API 가용성이 마비될 수 있습니다.
* **진단 기준**: 설정된 임계치 이상의 API 요청 발생 시 자동으로 접속이 차단되거나 제어되어야 합니다.
* **진단 방법(예시)**: `ab (ApacheBench)` 툴을 이용해 초당 100회 이상의 가상 요청을 전송한 후 서버가 `429 Too Many Requests`를 반환하는지 확인합니다.
* **기반시설기준항목**: 서비스 관리 - DoS 부하 경감 대책

## 2. Infrastructure & Network 보안

### 🧱 Proxmox 가상화 및 내부 방화벽 보안
* **설명**: Proxmox 내부 가상화 자원(VM/LXC) 간의 무분별한 횡적 이동(Lateral Movement)을 차단합니다.
* **설정 방법**: Proxmox Datacenter -> Firewall 메뉴를 활성화하고, 각 VM/LXC의 네트워크 인터페이스에 꼭 필요한 포트(예: 22, 6443 등)만 허용하는 `Default Drop` 정책을 적용합니다.
* **공격 가능성**: 하나의 가상머신(VM)이 악성코드에 감염되었을 때, 동일한 브릿지 네트워크 내의 다른 모든 서버 및 개발 인프라가 차단 없이 공격받을 수 있습니다.
* **진단 기준**: 비인가된 IP 및 포트로부터의 인바운드 트래픽이 방화벽 단에서 차단되어야 합니다.
* **진단 방법(예시)**: 인가되지 않은 외부 VM에서 타 VM을 대상으로 `nmap -p 1-65535 [타겟_IP]` 스캔을 수행하여 필수 포트 외에는 `filtered` 혹은 `closed` 상태인지 확인합니다.
* **기반시설기준항목**: 네트워크 보안 - 방화벽 설정 및 접근 통제

### 🕸️ Kubernetes Cluster 통신 보안 (Network Policy)
* **설명**: K8s 내부 파드(Pod) 간 통신을 제어하여 격리 수준을 높입니다.
* **설정 방법**: Calico 등의 CNI 플러그인을 활용해 `NetworkPolicy` 오브젝트를 생성합니다. DB 파드에는 인바운드 규칙으로 오직 백엔드 앱 파드의 라벨(`app: backend`)만 허용하도록 지정합니다.
* **공격 가능성**: 웹 애플리케이션 파드가 침해당했을 때, 공격자가 K8s 내부 네트워크를 통해 격리되지 않은 DB 파드에 직접 접근하여 데이터를 탈취할 수 있습니다.
* **진단 기준**: 네임스페이스 및 파드 간 통신이 인가된 경로 외에는 차단되어야 합니다.
* **진단 방법(예시)**: 임의의 테스트 파드를 생성한 뒤 DB 파드의 IP로 `nc -zv [DB_Pod_IP] [DB_Port]`를 실행하여 접속이 타임아웃(차단)되는지 확인합니다.
* **기반시설기준항목**: 네트워크 보안 - 네트워크 분리 및 접근 제어

## 3. OS (Linux) & CI/CD 노드 보안

### 🚫 Root 원격 접속 제한
* **설명**: Proxmox 노드 자체 및 내부 VM, Jenkins, K8s Worker Node에 root 계정으로 원격 로그인하는 것을 전면 금지합니다.
* **설정 방법**: `/etc/ssh/sshd_config` 파일에서 `PermitRootLogin no`로 수정하고 SSH 서비스를 재시작합니다.
* **공격 가능성**: 공격자가 root 계정을 대상으로 무차별 대입 공격을 성공할 경우 시스템의 최고 권한을 즉시 확보하게 됩니다.
* **진단 기준**: SSH를 통한 외부 root 로그인이 거부되어야 합니다.
* **진단 방법(예시)**: 외부 터미널에서 `ssh root@[서버_IP]` 접속 시 `Permission denied` 에러가 발생하며 접속이 차단되는지 확인합니다.
* **기반시설기준항목**: 시스템 보안 - Root 계정 원격 접속 제한

### 🔑 SSH 키 기반 인증 적용
* **설명**: 패스워드 인증 방식을 폐기하고 비대칭키(공개키/개인키) 쌍을 이용한 인증만 허용합니다.
* **설정 방법**: `/etc/ssh/sshd_config` 파일 내 `PasswordAuthentication no` 및 `PubkeyAuthentication yes`를 설정합니다.
* **공격 가능성**: 단순한 패스워드를 사용할 경우 사전 공격(Dictionary Attack)이나 무차별 대입 공격에 의해 쉽게 무력화될 수 있습니다.
* **진단 기준**: SSH 접속 시 패스워드 입력창이 뜨지 않고 오직 등록된 SSH Key 파일로만 로그인이 가능해야 합니다.
* **진단 방법(예시)**: SSH 키가 없는 환경에서 `ssh -o PubkeyAuthentication=no user@[서버_IP]`로 접속을 시도하여 인증 실패 처리가 되는지 확인합니다.
* **기반시설기준항목**: 시스템 보안 - 사용자 인증 및 원격 접근 제어

### 🔐 패스워드 복잡성 설정
* **설명**: 시스템 계정(일반 유저) 생성 및 변경 시 안전한 비밀번호 규칙을 강제합니다.
* **설정 방법**: `/etc/security/pwquality.conf` 파일에서 `minlen = 12`, `dcredit = -1`, `ucredit = -1`, `lcredit = -1`, `ocredit = -1` 등을 설정하여 영문 대소문자, 숫자, 특수문자 조합을 의무화합니다.
* **공격 가능성**: 유추하기 쉬운 비밀번호를 사용할 경우 내부 인사이더 공격이나 2차 침투 시 쉽게 권한을 탈취당합니다.
* **진단 기준**: 취약한 패스워드로 변경을 시도할 때 시스템에서 거부 메시지가 출력되어야 합니다.
* **진단 방법(예시)**: 일반 계정으로 로그인 후 `passwd` 명령을 통해 `1234` 또는 `password`와 같은 단순한 문자열로 변경을 시도하여 실패하는지 확인합니다.
* **기반시설기준항목**: 시스템 보안 - 계정 패스워드 복잡성 설정

### ⚙️ IP Forwarding 제어 (커널 파라미터 보안)
* **설명**: 라우터 역할을 하지 않는 일반 노드나 VM에서 패킷 포워딩 기능을 비활성화하여 허브 역할을 방지합니다.
* **설정 방법**: `/etc/sysctl.conf` 파일에 `net.ipv4.ip_forward = 0` (K8s Master/Worker 노드 및 라우팅 VM 제외)을 명시하고 `sysctl -p`를 적용합니다.
* **공격 가능성**: 악의적인 사용자가 해당 서버를 중간 경유지로 삼아 내부망의 다른 자원으로 공격 패킷을 우회 전달할 수 있습니다.
* **진단 기준**: 라우팅 목적 외 노드들의 커널 파라미터 값이 0으로 설정되어 있어야 합니다.
* **진단 방법(예시)**: 터미널에서 `sysctl net.ipv4.ip_forward` 명령을 실행하여 결과 값이 `0`인지 확인합니다.
* **기반시설기준항목**: SYSTEM 보안 - 불필요한 서비스 및 기능 제한

## 4. WAS & Container 보안 (AWS 마이그레이션 대비)

### 👤 실행 계정 권한 제한 (Root 실행 금지)
* **설명**: Tomcat, Node.js, Python, Jenkins, Gitlab 등 모든 어플리케이션을 OS의 root가 아닌 전용 일반 유저 계정으로 실행합니다. Harbor를 사용할 경우 Dockerfile 빌드 시에도 일반 유저 설정을 적용합니다.
* **설정 방법**: 
  * OS 레벨: `useradd -s /sbin/nologin appuser`와 같이 로그인 권한이 없는 계정을 생성한 뒤 `chown`으로 권한을 인계하고 실행합니다.
  * Dockerfile 레벨: 배포할 컨테이너 이미지 빌드 시 마지막 레이어 부근에 `USER 1001` 또는 `USER appuser` 형식을 명시합니다.
* **공격 가능성**: 애플리케이션의 취약점(RCE 등)으로 인해 쉘이 획득되었을 때, 프로세스가 root로 실행 중이라면 공격자가 즉시 호스트 시스템 전체 제어권을 장악하게 됩니다.
* **진단 기준**: 실행 중인 WAS 프로세스의 소유자(UID)가 0(root)이 아니어야 합니다.
* **진단 방법(예시)**: 구동 중인 컨테이너 내부나 서버 호스트에서 `ps -ef | grep tomcat` 또는 `whoami`를 실행하여 일반 사용자 계정명이나 UID가 노출되는지 확인합니다.
* **기반시설기준항목**: 서비스 관리 - 애플리케이션 프로세스 권한 제한

### 🛑 에러 페이지 정보 노출 방지 및 배너 숨김
* **설명**: 시스템 장애 시 나타나는 상세 에러 내용(StackTrace, 프레임워크 버전, 인프라 구조 정보)과 서버 헤더 정보를 숨깁니다.
* **설정 방법**: 
  * Jenkins/Harbor: 기본 에러 페이지 설정을 변경하여 사용자에게 내부 로그 대신 커스텀 안내 페이지(500, 404)를 반환하도록 대체합니다.
  * Ingress/Nginx: `nginx.ingress.kubernetes.io/server-snippet: "server_tokens off;"` 옵션이나 전역 ConfigMap 설정을 통해 HTTP 응답 헤더에서 `Server: nginx/1.21.x` 구조의 버전 표기를 제거합니다.
* **공격 가능성**: 에러 메시지에 노출된 특정 소프트웨어 버전을 바탕으로 공격자가 알려진 취약점(CVE)을 매핑하여 타겟형 정밀 공격을 수행할 수 있습니다.
* **진단 기준**: 404, 500 등의 에러 발생 시 시스템 버전 정보 및 소스코드 디버깅 정보가 외부 화면에 표시되지 않아야 합니다.
* **진단 방법(예시)**: 존재하지 않는 경로(`example.com`)를 고의로 호출한 후 반환되는 응답 바디와 헤더에 버전 정보나 시스템 구조가 포함되어 있는지 검증합니다.
* **기반시설기준항목**: 서비스 관리 - 웹 서비스 디렉터리 및 정보 노출 제한

### 📦 컨테이너 이미지 경량화 적용
* **설명**: Harbor에 업로드 및 배포할 Dockerfile 빌드 시 불필요한 도구와 패키지를 배제하고 필요한 최소 파일만 포함합니다.
* **설정 방법**: `ubuntu`나 `centos` 기반의 무거운 베이스 이미지 대신 `alpine` 혹은 구글의 `distroless` 이미지를 베이스로 사용하고, 멀티 스테이지 빌드(Multi-stage build) 기법을 도입하여 빌드 도구(`gcc`, `maven` 등)를 최종 이미지에서 제외합니다.
* **공격 가능성**: 이미지 내부에 `curl`, `wget`, `netcat`, `tar` 등의 유틸리티가 남아 있으면, 컨테이너 침투 성공 시 공격자가 외부에서 악성 스크립트를 다운로드하고 추가 툴을 실행하기 매우 수월해집니다. 또한 이미지 크기가 커질수록 보안 취약점(CVE) 개수가 비례하여 증가합니다.
* **진단 기준**: 최종 프로덕션 이미지 내에 컴파일러나 불필요한 네트워크 유틸리티 패키지가 설치되어 있지 않아야 합니다.
* **진단 방법(예시)**: Harbor 내 내장된 취약점 스캐너(Trivy 등) 결과 보고서를 조회하여 Critical/High 등급의 패키지 취약점 개수가 최소화되었는지 점검합니다.
* **기반시설기준항목**: 서비스 관리 - 서버 소프트웨어 최신 패치 및 보안 최적화

## 5. DB 보안

### 🚫 DB 원격 접속 제한
* **설명**: Percona XtraDB Cluster(PXC), MariaDB Cluster 서버로의 다이렉트 외부 인터넷 접속을 전면 차단합니다.
* **설정 방법**: DB 구성 파일(`/etc/mysql/my.cnf`) 내 `bind-address = 127.0.0.1` 또는 사설 대역 IP(`10.x.x.x`)로 지정하고, 외부 공인 IP에서의 3306 포트 접근은 방화벽(AWS Security Group 또는 Proxmox Firewall)에서 차단합니다.
* **공격 가능성**: DB 포트가 외부에 노출되어 있으면 지속적인 Credential Stuffing 및 무차별 대입 공격을 받게 되며, 제로데이 취약점 발생 시 데이터베이스가 즉시 통째로 유출되거나 랜섬웨어에 감염될 수 있습니다.
* **진단 기준**: 외부 공인 네트워크 환경에서 DB 포트로의 직접적인 TCP 핸드셰이크가 실패해야 합니다.
* **진단 방법(예시)**: 사외 네트워크망 PC 터미널에서 `telnet [DB_공인_IP] 3306` 혹은 `mysql -h [DB_공인_IP]` 명령을 실행하여 접속이 거부되거나 타임아웃 되는지 확인합니다.
* **기반시설기준항목**: 데이터베이스 보안 - 데이터베이스 원격 접근 제한

### 🗂️ 계정별 권한 분리 (Principle of Least Privilege)
* **설명**: 하나의 관리자 계정(`root`)을 공유하지 않고, 애플리케이션 서비스 계정과 DBA 관리자 계정을 엄격히 분리하여 최소 권한만 부여합니다.
* **설정 방법**: 앱 전용 계정 생성 시 `GRANT SELECT, INSERT, UPDATE, DELETE ON dbname.* TO 'app_user'@'10.x.x.x';`와 같이 구조 변경(`DROP`, `ALTER`) 권한을 배제하고 특정 사설 IP 대역에서만 접근할 수 있도록 제한합니다.
* **공격 가능성**: 웹 애플리케이션의 SQL Injection 취약점 등으로 계정 권한이 탈취되었을 때, 해당 계정이 `SUPER` 또는 `root` 권한을 가지고 있다면 데이터베이스 전체 삭제(Drop)나 시스템 명령 실행(`sys_eval`)으로 이어질 수 있습니다.
* **진단 기준**: 각 애플리케이션이 사용하는 DB 계정이 불필요한 DDL/DCL 권한 및 타 데이터베이스 스키마 조회 권한을 가지지 않아야 합니다.
* **진단 방법(예시)**: 애플리케이션 설정에 명시된 계정으로 DB에 직접 로그인한 뒤 `SHOW DATABASES;`를 수행하여 타 서비스의 DB가 보이는지 확인하고, `DROP TABLE` 명령을 시험 구동하여 권한 거부(`Access denied`)가 발생하는지 확인합니다.
* **기반시설기준항목**: 데이터베이스 보안 - 데이터베이스 사용자 권한 제어

### 🔒 TLS 클러스터간 통신 암호화 적용
* **설명**: PXC 및 MariaDB Galera Cluster 노드 간에 상태 전송(SST, IST) 및 데이터 동기화 복제를 진행할 때 패킷을 암호화합니다.
* **설정 방법**: 각 DB 노드에 SSL 인증서(CA, Server Cert, Private Key)를 생성 및 배치하고, 설정 파일 내 `wsrep_provider_options="socket.ssl_key=...; socket.ssl_cert=...; socket.ssl_ca=...;"` 및 `ssl-mode=REQUIRED` 옵션을 명시합니다.
* **공격 가능성**: 데이터 복제가 일어나는 가상화 네트워크 구간에서 스니핑이 발생할 경우, 노드 간 주고받는 세션 정보 및 테이블 내 고유 식별 정보, 민감 데이터 평문이 그대로 노출됩니다.
* **진단 기준**: 클러스터 노드 간의 동기화 트래픽 및 클라이언트-서버 간 통신 시 SSL/TLS 사용 플래그가 활성화되어 있어야 합니다.
* **진단 방법(예시)**: DB 세션 접속 후 `SHOW STATUS LIKE 'Ssl_cipher';` 또는 `SHOW VARIABLES LIKE 'have_ssl';`을 수행하여 결과 값이 활성화(`YES` 또는 적용된 암호화 알고리즘명) 상태인지 검증합니다.
* **기반시설기준항목**: 데이터베이스 보안 - 데이터베이스 암호화 통신 적용
