# Percona XtraDB Cluster 전체 이해 문서

## 1. 프로젝트 개요

### 무엇을 만들려고 하는가?

**목표**: Proxmox 가상화 환경에서 Percona XtraDB Cluster(PXC)를 구축하여 고가용성(HA) MySQL 데이터베이스 클러스터 만들기

**왜 필요한가?**
- 데이터베이스 장애 시 서비스 중단 방지
- 여러 서버에 데이터를 복제하여 데이터 손실 방지
- 로드밸런싱으로 읽기/쓰기 성능 향상
- Proxmox HA로 VM 장애 시 자동 복구
- DMZ/내부망 분리로 보안 강화

**구성 방식**:
- Kubernetes 사용 안 함 → VM에 직접 설치
- Terraform으로 VM 생성
- Ansible로 소프트웨어 설치 및 설정
- Proxmox HA로 VM 고가용성 보장
- DMZ와 내부망 분리 (보안)

## 2. 전체 아키텍처 설명

### 2-1. 네트워크 분리 구조

```
┌─────────────────────────────────────────────────────────────────┐
│                        외부 네트워크                             │
│                    (애플리케이션, 사용자)                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         DMZ (VLAN 20)                           │
│                        172.16.20.0/24                           │
│                        Gateway: 172.16.20.1                     │
│         (외부 접근 가능, HAProxy/ProxySQL 배치)                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ (pfSense Firewall)
                              │ DMZ → Internal: Allow MySQL
                              │ Internal → DMZ: Block
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      내부망 (VLAN 30)                            │
│                       172.16.30.0/24                            │
│                       Gateway: 172.16.30.1                      │
│         (외부 접근 불가, Percona 데이터베이스 배치)               │
└─────────────────────────────────────────────────────────────────┘
```

**DMZ (172.16.20.x)**:
- 외부에서 접근 가능한 서비스 배치
- HAProxy, ProxySQL: 로드밸런서, 프록시
- 애플리케이션 연결 포인트

**내부망 (172.16.30.x)**:
- 외부 접근 불가 (보안)
- Percona 데이터베이스만 배치
- HAProxy/ProxySQL만 접근 가능

### 2-2. 물리적 구조

```
Proxmox 클러스터 (4개 서버)
├── pve1 (Proxmox 노드 1)
├── pve2 (Proxmox 노드 2)
├── pve3 (Proxmox 노드 3)
└── pve4 (Proxmox 노드 4)
```

**Proxmox 역할**:
- 가상화 플랫폼 (VM 실행)
- 클러스터 구성 (4개 노드가 하나로 연동)
- HA (High Availability): 노드 장애 시 VM 자동 마이그레이션
- 스토리지: local-lvm (로컬 디스크)

### 2-2. 가상화 구조 (VM 배치)

```
각 Proxmox 노드에 VM 분산 배치:

pve1:
  ├── pxc-1 (Percona DB 노드 1) - 172.16.30.10
  └── haproxy-1 (HAProxy LB 1) - 172.16.20.20

pve2:
  ├── pxc-2 (Percona DB 노드 2) - 172.16.30.11
  └── haproxy-2 (HAProxy LB 2) - 172.16.20.21

pve3:
  ├── pxc-3 (Percona DB 노드 3) - 172.16.30.12
  └── proxysql-1 (ProxySQL 1) - 172.16.20.25

pve4:
  └── proxysql-2 (ProxySQL 2) - 172.16.20.26
```

**분산 배치 이유**:
- 한 노드 장애 시 모든 VM이 같이 장애나지 않도록 분산
- HA 그룹으로 VM 자동 마이그레이션 가능

### 2-4. 네트워크 구조

```
pfSense (방화벽/라우터)
    ├── VLAN 20: DMZ (외부 접근 가능)
    │   ├── HAProxy: 172.16.20.20-21
    │   ├── ProxySQL: 172.16.20.25-26
    │   └── VIPs: 172.16.20.30, 172.16.20.35
    │
    ├── VLAN 30: 내부망 (외부 접근 불가)
    │   ├── Percona: 172.16.30.10-12
    │   └── Gateway: 172.16.30.1
    │
    └── 방화벽 규칙
        ├── 외부 → DMZ: Allow (3306, 3307, 6033)
        ├── DMZ → 내부망: Allow MySQL (3306)
        ├── 내부망 → DMZ: Block
        └── 외부 → 내부망: Block
```

**pfSense 역할**:
- VLAN 생성 및 관리 (VLAN 20, 30)
- VLAN 간 라우팅
- 방화벽 규칙 적용 (DMZ ↔ 내부망 통신 제어)

**VLAN 20 (DMZ)**:
- 외부 접근 가능
- HAProxy, ProxySQL 배치
- 애플리케이션 연결 포인트

**VLAN 30 (내부망)**:
- 외부 접근 불가 (보안)
- Percona 데이터베이스만 배치
- DMZ에서만 접근 가능 (MySQL 포트)

## 3. 각 구성 요소 역할 설명

### 3-1. Percona XtraDB Cluster (PXC)

**무엇인가?**: MySQL 클러스터 솔루션 (Galera 기반)

**주요 특징**:
- **동기 복제**: 모든 노드에 데이터 동시写入
- **다중 마스터**: 모든 노드에서 읽기/쓰기 가능
- **자동 노드 추가**: 새 노드가 자동으로 데이터 동기화
- **장애 자동 감지**: 노드 장애 시 자동으로 클러스터 재구성

**구성**:
```
3개 노드 (pxc-1, pxc-2, pxc-3)
    
pxc-1 (Bootstrap 노드):
  ├── 클러스터 초기화
  ├── 첫 번째 시작 시 "bootstrap" 모드
  └── 다른 노드가 이 노드에 연결
      
pxc-2, pxc-3:
  ├── pxc-1에 연결하여 클러스터 가입
  ├── SST (State Snapshot Transfer)로 데이터 복사
  └── 동기화 후 일반 노드로 작동
```

**Galera Replication 작동 방식**:
```
읽기 쿼리: 각 노드에서 로컬 처리
      ↓
쓰기 쿼리:
  ├── 노드 1에서 INSERT 실행
  ├── Certification (모든 노드에서 동시 검증)
  ├── 모든 노드에 Apply (동시写入)
  └── 모든 노드에 Commit (동시 완료)
```

**장점**:
- 노드 장애 시 데이터 손실 없음
- 읽기 성능 향상 (모든 노드에서 읽기 가능)
- 자동 복구

**단점**:
- 쓰기 성능은 단일 노드와 비슷 (모든 노드에写入 필요)
- 네트워크 지연 시 쓰기 성능 저하
- 모든 노드가 동일한 데이터 (스케일 아웃 제한)

### 3-2. HAProxy

**무엇인가?**: TCP/HTTP 로드밸런서

**역할**:
- MySQL 연결 분산
- 헬스체크 (노드 상태 확인)
- 읽기/쓰기 분리

**구성**:
```
HAProxy 구성:
    ├── Port 3306: 읽기 쿼리 분산
    │     └── Round Robin (순차 분산)
    │     └── 모든 Percona 노드로 분산
    │
    └── Port 3307: 쓰기 쿼리 분산
    │     └── Least Connections (가장 적은 연결)
    │     └── 단일 노드로集中
    │
    └── Port 8404: Stats Interface
    │     └── 웹 UI로 상태 확인
```

**작동 방식**:
```
애플리케이션 → HAProxy (172.16.20.30:3306)
                ├── pxc-1 연결 → 읽기 쿼리 처리
                ├── pxc-2 연결 → 읽기 쿼리 처리
                └── pxc-3 연결 → 읽기 쿼리 처리
                
애플리케이션 → HAProxy (172.16.20.30:3307)
                ├── 가장 적은 연결의 노드 선택
                └── 쓰기 쿼리 처리
```

**헬스체크**:
```
HAProxy가 각 Percona 노드 확인:
  ├── TCP로 3306 포트 체크
  ├── 노드 응답 없으면 "DOWN" 표시
  └── DOWN 노드는 트래픽 분산에서 제외
```

**VIP (Virtual IP)**: 
- `172.16.20.30` - HAProxy VIP
- 애플리케이션은 이 IP로 연결
- VIP를 사용하는 HAProxy 노드가 실제 트래픽 처리
- HAProxy 노드 장애 시 VIP가 다른 노드로 이동 (Keepalived 필요)

### 3-3. ProxySQL

**무엇인가?**: MySQL 프록시 (쿼리 라우팅)

**역할**:
- 쿼리 분석 및 라우팅
- 커넥션 풀링
- 쿼리 캐싱
- 읽기/쓰기 자동 분리

**구성**:
```
ProxySQL 구성:
    ├── Port 6032: Admin Interface
    │     └── 설정, 모니터링
    │     └── mysql 명령어로 관리
    │
    └── Port 6033: MySQL Proxy
    │     └── 애플리케이션 연결 포트
    │     └── 쿼리 분석 후 적절한 노드로 라우팅
```

**쿼리 라우팅**:
```
ProxySQL 작동 방식:
    
읽기 쿼리 (SELECT):
  ├── Hostgroup 10 (모든 노드)
  ├── Round-robin 분산
  └── 읽기 성능 향상
      
쓰기 쿼리 (INSERT, UPDATE, DELETE):
  ├── Hostgroup 0 (단일 writer)
  ├── 모든 쓰기集中
  └── 일관성 보장
```

**커넥션 풀링**:
```
ProxySQL이 MySQL 연결 관리:
    
애플리케이션 → ProxySQL (연결 요청)
                ├── Pool에서 기존 연결 사용
                ├── 새 연결 생성 불필요
                └── 연결 생성 overhead 감소
                
ProxySQL → Percona (실제 MySQL 연결)
                ├── 미리 생성된 연결 Pool
                ├── 연결 재사용
                └── Percona 부하 감소
```

**쿼리 캐싱**:
```
ProxySQL이 쿼리 결과 캐싱:
    
SELECT 쿼리:
  ├── ProxySQL에서 캐싱
  ├── 동일 쿼리 → 캐시된 결과 반환
  └── Percona 쿼리 불필요
```

**장점 vs HAProxy**:
- ProxySQL: 쿼리 분석, 캐싱, 커넥션 풀
- HAProxy: TCP 레벨 로드밸런싱만

### 3-4. Terraform

**무엇인가?**: 인프라 자동화 도구 (Infrastructure as Code)

**역할**:
- Proxmox에서 VM 생성
- VM 리소스 설정 (CPU, Memory, Disk)
- 네트워크 설정 (VLAN, IP)
- HA 설정
- Ansible 인벤토리 파일 생성

**작동 방식**:
```
Terraform 실행 과정:
    
1. terraform init
   └── Provider (Proxmox API) 초기화
   
2. terraform plan
   └── 변경 사항 확인 (VM 생성 계획)
   
3. terraform apply
   ├── Proxmox API로 VM 생성 요청
   ├── VM 템플릿에서 클론
   ├── CPU/Memory/Disk 설정
   ├── 네트워크 설정 (VLAN 30)
   ├── Cloud-init으로 IP 설정
   ├── HA 그룹에 VM 추가
   └── Ansible 인벤토리 파일 생성
```

**모듈 구조**:
```
terraform/
├── modules/
│   ├── proxmox-vm/
│   │   ├── main.tf      → VM 리소스 정의
│   │   ├── variables.tf → 입력 변수 정義
│   │   └── outputs.tf   → 출력 정보
│   │
│   └── percona-cluster/
│   │   ├── main.tf      → VM 그룹 생성 로직
│   │   ├── variables.tf → 클러스터 변수
│   │   └── outputs.tf   → 클러스터 정보
│
└── environments/
    └── production/
        ├── main.tf              → Provider + 모듈 호출
        ├── variables.tf         → 환경 변수
        ├── terraform.tfvars     → 실제 값 입력
        └── templates/
            └── inventory.ini.tftpl → Ansible 인벤토리 템플릿
```

**모듈 vs 환경**:
- **모듈**: 재사용 가능한 구성 단위
- **환경**: 실제 배포 설정 (dev, prod 등)

### 3-5. Ansible

**무엇인가?**: 구성 관리 자동화 도구 (Configuration as Code)

**역할**:
- VM에 소프트웨어 설치
- Percona XtraDB Cluster 구성
- HAProxy 설정
- ProxySQL 설정
- 클러스터 부트스트랩

**작동 방식**:
```
Ansible 실행 과정:
    
1. SSH로 VM 접속
   └── terraform이 생성한 inventory 파일 사용
   
2. Playbook 실행
   ├── Tasks 순차 실행
   ├── 조건문, 반복문 사용
   └── 롤(Role) 단위로 구성
      
3. 소프트웨어 설치
   ├── apt로 패키지 설치
   ├── 템플릿으로 설정 파일 생성
   └── 서비스 시작
```

**롤(Role) 구조**:
```
ansible/
├── roles/
│   ├── common/
│   │   ├── tasks/main.yml
│   │   └── 기본 패키지 설치, 시스템 설정
│   │
│   ├── percona/
│   │   ├── tasks/main.yml       → PXC 설치 로직
│   │   ├── templates/my.cnf.j2  → MySQL 설정 템플릿
│   │   ├── defaults/main.yml    → 기본 변수
│   │   ├── handlers/main.yml    → 서비스 재시작
│   │   └── Percona 설치, Galera 구성, 클러스터 가입
│   │
│   ├── haproxy/
│   │   ├── tasks/main.yml
│   │   ├── templates/haproxy.cfg.j2
│   │   ├── HAProxy 설치, 로드밸런싱 설정
│   │
│   └── proxysql/
│   │   ├── tasks/main.yml
│   │   ├── templates/proxysql.cnf.j2
│   │   ├── ProxySQL 설치, 쿼리 라우팅 설정
│
├── playbooks/
│   ├── site.yml     → 전체 배포 플레이북
│   └── verify.yml   → 검증 플레이북
│
└── inventory/
    ├── hosts.ini    → VM 목록 (terraform이 생성)
    └── group_vars/
        └── all.yml  → 전역 변수
```

**Playbook 실행 순서**:
```
site.yml:
    
1. all 그룹 (모든 VM)
   └── common 롤
   └── 기본 패키지 설치, 호스트 설정
   
2. haproxy 그룹
   └── haproxy 롤
   └── HAProxy 설치, 로드밸런싱 설정
   
3. proxysql 그룹
   └── proxysql 롤
   └── ProxySQL 설치, 쿼리 라우팅 설정
   
4. bootstrap 그룹 (pxc-1)
   └── percona 롤 (pxc_bootstrap=true)
   └── 클러스터 부트스트랩 (최초 시작)
   
5. percona 그룹 (pxc-2, pxc-3)
   └── percona 롤 (pxc_bootstrap=false)
   └── 클러스터 가입 (기존 클러스터에 연결)
```

## 4. 전체 배포 과정 (Step by Step)

### Step 0: 사전 준비

**필요한 것**:
1. Proxmox 클러스터 (4개 노드)
2. pfSense (VLAN 30 구성)
3. Ubuntu 템플릿 VM (cloud-init 포함)
4. Terraform, Ansible 설치된 관리 PC
5. Proxmox API Token

**Ubuntu 템플릿 준비**:
```bash
# Proxmox에서 Ubuntu VM 생성
# cloud-init 설정
# 템플릿으로 변환
# 이름: ubuntu-22.04-template
```

**pfSense VLAN 30 설정**:
```
1. Interfaces → Assignments → VLANs
   ├── Parent Interface: vtnet1 (LAN)
   ├── VLAN Tag: 30
   └── Description: Database_Network
   
2. Interfaces → Assignments
   └── vtnet1.30 추가
   
3. Interfaces → [VLAN30]
   ├── Enable: ☑
   ├── IPv4: Static
   └── IP: 172.16.30.1/24
   
4. Firewall → Rules → [VLAN30]
   └── Pass Any Any (테스트용, 추후 제한)
```

**Proxmox API Token 생성**:
```bash
# Proxmox 웹 UI
Datacenter → Permissions → API Tokens
  ├── User: root@pam
  ├── Add API Token
  ├── ID: terraform
  └── Secret 저장
```

### Step 1: Terraform 설정

**terraform.tfvars 작성**:

```bash
cd terraform/environments/production
```


### Step 2: Terraform 실행

**초기화**:

## 5. 기술 스택 상세 설명

### 5-1. Galera Replication

**무엇인가**: 동기 복제 라이브러리 (PXC의 핵심)

**작동 방식**:
```
쓰기 작업 흐름:

노드 A에서 INSERT 실행:
  ├── 노드 A: 쿼리 실행
  ├── Write Set 생성 (변경 내역)
  ├── 모든 노드에 Write Set 전송
  ├── Certification (모든 노드에서 동시 검증)
  │     ├── Primary Key 체크
  │     ├── Unique Constraint 체크
  │     ├── Conflict 없으면 Apply
  ├── 모든 노드에 Apply (실제 데이터 변경)
  ├── 모든 노드에 Commit
  └── 노드 A: 결과 반환
```

**Certification 작동**:
```
두 노드에서 동시에 같은 레코드 수정:

노드 A: UPDATE users SET name='A' WHERE id=1
노드 B: UPDATE users SET name='B' WHERE id=1

Write Set 전송:
  ├── Certification 순서 결정 (Global Order)
  ├── 먼저 도착한 Write Set: Apply
  ├── 나중에 도착한 Write Set: Reject
  └── Reject된 노드: 에러 반환, Rollback
```

**SST (State Snapshot Transfer)**:
```
새 노드 가입 시 데이터 전송:

노드 D가 클러스터 가입:
  ├── 클러스터에서 Donor 노드 선택
  ├── Donor 노드의 전체 데이터 스냅샷 생성
  ├── xtrabackup으로 데이터 복사
  ├── 노드 D로 전송
  ├── 노드 D: 데이터 로드
  └── 노드 D: 클러스터 동기화 완료
```

**IST (Incremental State Transfer)**:
```
노드가 잠시 분리 후 재가입:

노드 C가 재가입:
  ├── 노드 C: "나는 seq 1000까지 있음"
  ├── 클러스터: "현재 seq 1100"
  ├── seq 1001-1100 Write Set만 전송
  ├── 노드 C: 누락된 변경만 Apply
  └── 동기화 완료 (SST 불필요)
```

### 5-2. Percona XtraDB Cluster 특징

**vs MySQL Replication**:
```
MySQL Replication (비동기):
  ├── Master에서 쓰기
  ├── Slave에 비동기 복제
  ├── Slave 지연 가능
  ├── Master 장애 시 데이터 손실 가능
  └── 읽기 분산, 쓰기集中

Percona XtraDB Cluster (동기):
  ├── 모든 노드에서 쓰기
  ├── 동기 복제 (모든 노드에立即写入)
  ├── 지연 없음
  ├── 노드 장애 시 데이터 손실 없음
  └── 모든 노드에서 읽기/쓰기 가능
```

**vs MySQL Group Replication**:
```
MySQL Group Replication:
  ├── MySQL 5.7+ 내장
  ├── 동기 복제
  ├── Single Primary 모드 (쓰기集中)
  └── Multi Primary 모드 (모든 노드 쓰기)

Percona XtraDB Cluster:
  ├── Galera 기반
  ├── 동기 복제
  ├── 모든 노드 Multi-Master
  ├── 더 성숙한 기술
  └── 더 많은 실무 경험
```

### 5-3. HAProxy 로드밸런싱 방식

**Round Robin**:
```
읽기 쿼리 분산:

연결 1 → pxc-1
연결 2 → pxc-2
연결 3 → pxc-3
연결 4 → pxc-1 (다시 순환)
...
```

**Least Connections**:
```
쓰기 쿼리 분산:

pxc-1: 5 connections
pxc-2: 3 connections ← 선택
pxc-3: 7 connections

새 연결 → pxc-2 (가장 적은 연결)
```

**Health Check**:
```
HAProxy가 노드 상태 확인:

TCP Check:
  ├── HAProxy → pxc-1:3306 (TCP 연결)
  ├── 응답 OK → UP 상태
  ├── 응답 없음 → DOWN 상태
  └── DOWN 노드는 분산에서 제외

MySQL Check (더 정밀):
  ├── MySQL 프로토콜로 헬스체크
  ├── wsrep_local_state 확인
  ├── Synced만 UP
  └── Donor/Joining은 DOWN
```

### 5-4. ProxySQL 쿼리 라우팅

**쿼리 패턴 매칭**:
```
ProxySQL이 쿼리 분석:

SELECT * FROM users:
  ├── match_pattern: "^SELECT"
  ├── destination_hostgroup: 10
  └── 읽기 hostgroup으로 라우팅

INSERT INTO users:
  ├── match_pattern: "^INSERT"
  ├── destination_hostgroup: 0
  └── 쓰기 hostgroup으로 라우팅
```

**Hostgroup 구성**:
```
Hostgroup 0 (Writer):
  ├── pxc-1
  ├── 모든 쓰기 쿼리集中
  └── 단일 노드 (일관성)

Hostgroup 10 (Reader):
  ├── pxc-1, pxc-2, pxc-3
  ├── 모든 읽기 쿼리 분산
  └── 모든 노드 사용
```

**쿼리 캐싱**:
```
ProxySQL이 쿼리 결과 캐싱:

SELECT * FROM users WHERE id=1:
  ├── Percona에서 실행
  ├── 결과 캐싱 (60초)
  
다음 동일 쿼리:
  ├── 캐시 확인
  ├── 캐시 있음 → 캐시된 결과 반환
  └── Percona 쿼리 불필요
```

## 6. HA (High Availability) 설명

### 6-1. Proxmox HA

**HA 그룹**:
```
percona-ha 그룹:
  ├── Nodes: pve1, pve2, pve3, pve4
  ├── Restricted: yes (그룹 노드만 사용)
  └── VM을 그룹에 추가
```

**HA 설정값**:
```
max_relocate: 2
  ├── 노드 장애 시 VM 마이그레이션 최대 2회
  
max_restart: 2
  ├── VM 장애 시 재시작 최대 2회
  
state: started
  ├── VM이 실행 상태여야 함
```

**장애 시나리오**:
```
pve1 노드 장애:
  1. Proxmox HA Manager가 감지
  2. pve1의 VM을 다른 노드로 마이그레이션
     ├── pxc-1 → pve2
     ├── haproxy-1 → pve3
  3. VM이 새 노드에서 시작
  4. Percona 클러스터 자동 재구성
  5. 서비스 계속 작동
```

### 6-2. Percona HA

**노드 장애**:
```
pxc-1 장애:
  ├── 클러스터가 pxc-1을 감지
  ├── pxc-1을 클러스터에서 제외
  ├── pxc-2, pxc-3가 계속 작동
  ├── wsrep_cluster_size: 2
  ├── HAProxy가 pxc-1을 DOWN 표시
  └── 트래픽이 pxc-2, pxc-3로 분산
```

**클러스터 장애 (Split-Brain)**:
```
네트워크 분리로 노드 간 연결 끊김:

pxc-1, pxc-2: Primary (정상)
pxc-3: Non-Primary (분리됨)

pxc-3에서 쓰기 시도:
  ├── Certification 불가
  ├── 쿼리 에러
  └── Non-Primary 상태
  
pxc-3 복구:
  ├── 네트워크 연결 복원
  ├── IST로 동기화
  └── Primary로 재가입
```

### 6-3. HAProxy HA (Keepalived 필요)

**VIP 관리**:
```
현재 구성:
  ├── HAProxy VIP: 172.16.20.30
  ├── ProxySQL VIP: 172.16.20.35
  └── VIP를 사용하는 노드가 트래픽 처리
  
향후 Keepalived 추가:
  ├── haproxy-1: VIP MASTER
  ├── haproxy-2: VIP BACKUP
  ├── haproxy-1 장애 → VIP가 haproxy-2로 이동
  └── 트래픽 중단 없음
```

## 8. 문제 해결

### 8-1. Terraform 문제

**VM 생성 실패**:
```
Error: Unable to clone VM

확인:
  ├── Template 이름 확인
  ├── Storage 이름 확인
  ├── API Token 권한 확인
  └── Proxmox 노드 상태 확인
```

**IP 할당 실패**:
```
Error: IP address conflict

확인:
  ├── pfSense VLAN 30 설정
  ├── DHCP 비활성화 (고정 IP 사용)
  ├── IP 대역 확인 (172.16.30.x)
```

### 8-2. Ansible 문제

**SSH 연결 실패**:
```
Error: SSH connection refused

확인:
  ├── VM Cloud-init 완료 여부
  ├── SSH 키 일치 여부
  ├── ansible_user 확인
  └── VM IP 확인
```

**Percona 설치 실패**:
```
Error: Package not found

확인:
  ├── Percona Repository 활성화
  ├── apt update 실행
  └── Ubuntu 버전 확인 (22.04)
```

### 8-3. Percona 클러스터 문제

**노드 가입 실패**:
```
wsrep_local_state_comment: Joining

확인:
  ├── 네트워크 연결 (3306, 4444 포트)
  ├── SST 사용자 credentials
  ├── Donor 노드 상태
  └── Firewall 규칙
```

**Split-Brain**:
```
wsrep_cluster_status: Non-Primary

해결:
  1. 모든 MySQL 중지
     systemctl stop mysql
     
  2. Bootstrap 노드 선택
     # pxc-1에서
     systemctl start mysql
     # 또는
     galera_new_cluster
     
  3. 다른 노드 시작
     # pxc-2, pxc-3에서
     systemctl start mysql
```

### 8-4. HAProxy 문제

**Backend DOWN**:
```
HAProxy Stats에서 pxc-1이 DOWN

확인:
  ├── MySQL 프로세스 상태
  ├── 3306 포트 리스닝
  ├── Network connectivity
  └── Firewall
```

**Connection Refused**:
```
Error: Can't connect to MySQL server on '172.16.20.30'

확인:
  ├── HAProxy 프로세스 상태
  ├── HAProxy Port binding (3306, 3307)
  ├── HAProxy backend 설정
```

## 9. 성능 최적화

### 9-1. Percona 최적화

**my.cnf 설정**:
```ini
# innodb_buffer_pool_size
# 전체 Memory의 60-80%
innodb_buffer_pool_size = 3G  # 4GB RAM인 경우

# wsrep_slave_threads
# 복제 thread 수 (CPU core 수)
wsrep_slave_threads = 2

# innodb_flush_log_at_trx_commit
# 쓰기 성능 향상 (동기 복제이므로 0 가능)
innodb_flush_log_at_trx_commit = 0
```

### 9-2. HAProxy 최적화

**haproxy.cfg 설정**:
```
# Connection limits
maxconn 4000

# Timeout
timeout connect 5000
timeout client  50000
timeout server  50000

# Health check interval
option tcp-check
tcp-check connect port 3306
tcp-check connect timeout 1000
```

### 9-3. ProxySQL 최적화

**proxysql.cnf 설정**:
```
# Connection pooling
max_connections=2048

# Threads
threads=4

# Query cache
query_cache_size_MB=256
```

### 각 구성 요소 한 줄 설명

- **Terraform**: 인프라 자동화 (VM 생성)
- **Ansible**: 구성 자동화 (소프트웨어 설치)
- **Percona XtraDB Cluster**: MySQL 동기 복제 클러스터
- **Galera**: 동기 복제 라이브러리
- **HAProxy**: TCP 로드밸런서
- **ProxySQL**: MySQL 프록시 + 쿼리 라우팅
- **Proxmox HA**: VM 고가용성

### 최종 결과물

**VM 구성**:
- 3 Percona nodes (2 CPU, 4GB RAM, 60GB)
- 2 HAProxy nodes (2 CPU, 2GB RAM, 20GB)
- 2 ProxySQL nodes (2 CPU, 4GB RAM, 30GB)
- Total: 7 VMs, 14 CPU, 24GB RAM, 280GB Disk

**연결 방식**:
- HAProxy: 172.16.20.30:3306 (Read), :3307 (Write)
- ProxySQL: 172.16.20.35:6033 (Query Routing)
- Percona: 172.16.30.10-12:3306 (Direct)

**HA 보장**:
- Proxmox HA: 노드 장애 시 VM 마이그레이션
- Percona HA: 노드 장애 시 데이터 손실 없음
- HAProxy: 노드 장애 시 트래픽 분산 조정
